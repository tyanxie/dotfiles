# 编写设计文档（Spec）

## 目的

将需求分析阶段的结论转化为结构化的技术设计文档，作为后续实现的唯一参考依据。

## Spec 文档结构模板

```markdown
# [功能名称] 设计文档

## 概述

一段话说明该功能是什么、解决什么问题、面向谁。

## 设计目标

- 目标 1：...
- 目标 2：...
- 非目标：明确列出本次不做的事项

## 架构设计

整体架构描述，包括与现有系统的关系、新增组件的位置。
必要时附架构图（文本描述或 ASCII）。

## 模块职责

### 模块 A

- 职责描述
- 输入 / 输出
- 依赖关系

### 模块 B

- ...

## 接口定义

关键函数、类、API 的签名与语义说明。

## 数据流

描述数据从输入到输出的完整流转路径，标注各模块的处理节点。

## 错误处理

- 可预见的错误场景
- 对应的处理策略（重试、降级、抛出、日志）

## 边界条件

- 极端输入
- 并发 / 竞态
- 资源限制
- 兼容性约束
```

## 详略度指引

按模块复杂度调整各节的篇幅：

| 复杂度                               | 描述方式                           |
| ------------------------------------ | ---------------------------------- |
| 简单（纯粹的数据转换、薄封装）       | 几句话概括职责和接口即可           |
| 中等（有状态、有分支逻辑）           | 列出关键路径、状态转换、异常分支   |
| 复杂（涉及并发、外部依赖、性能敏感） | 详细展开，包含时序、约束、降级方案 |

原则：**读者能仅凭 spec 理解设计意图并开始实现**，不多不少。

## 保存路径

```
docs/powerflow/YYYY-MM-DD-feature-name/spec.md
```

- 日期为创建当天
- `feature-name` 使用 kebab-case，与需求分析文档同目录

## 工作流程

1. 按模板逐节撰写
2. 全部章节写完后统一展示给用户确认（复杂 spec 也可选择逐节确认）
3. 根据反馈修改，直到用户认可
4. 保存文件
5. 告知用户：spec 已保存至 `docs/powerflow/YYYY-MM-DD-feature-name/spec.md`，接下来进入 spec review loop

## TDD 可选提示

如果用户要求采用 TDD 方式开发，spec 中需额外体现可测试性考虑：

- **模块接口设计要便于单元测试**：公开接口语义明确、输入输出可断言
- **依赖可注入**：外部依赖（文件系统、网络、时间）通过参数或工厂注入，不硬编码
- **在接口定义节标注**：哪些接口是测试入口、哪些依赖需要 mock
- **在边界条件节补充**：哪些边界场景需要专门的测试用例覆盖
