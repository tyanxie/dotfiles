# 头脑风暴阶段

通过自然对话，将模糊想法转化为清晰的设计方案。

## 流程

### 1. 探索项目现状

- 查看相关文件、文档、近期提交
- 理解现有架构和模式
- 识别与需求相关的模块和接口

### 2. 逐个提问澄清需求

- **一次只问一个问题**
- **优先使用选择题**（给出 2-3 个选项 + 简要说明）
- 聚焦理解：目的、约束、成功标准
- 问题方向：
  - 功能边界（做什么、不做什么）
  - 用户场景（谁用、怎么用）
  - 技术约束（兼容性、性能、依赖）
  - 优先级（必须有 vs 可以有）

### 3. 提出方案

- 提出 **2-3 种方案**，每种包含：
  - 核心思路（一句话概括）
  - 优势
  - 劣势
  - 适用场景
- **必须给出倾向性建议**，并说明判断依据：
  - 需求复杂度（简单/中等/复杂）
  - 影响范围（局部/跨模块/全局）
  - 涉及模块数量
  - 维护成本

### 4. 确认推荐方案

- 明确推荐某一方案
- 解释为什么这个方案最适合当前场景
- 等待用户确认或调整

### 5. 询问后续路径

确认方案后，询问用户：

> 接下来你想：
> A) 走完整流程（编写详细 spec → 分步实现）
> B) 直接开发

- **必须给出倾向性建议**（A 或 B），并说明依据（需求复杂度、影响范围、涉及模块数）
- **选 A**：进入 spec 编写阶段（详见 [writing-spec.md](./writing-spec.md)）
- **选 B**：输出设计结论摘要后直接进入开发

## 设计结论摘要

当用户选择直接开发时，输出结构化摘要供后续引用：

```
## 设计结论

### 确认的功能点
- ...

### 约束条件
- ...

### 选定方案
- 方案名称：...
- 核心思路：...
- 关键设计决策：...
```

## 关键原则

- **一次一个问题**：不要一次抛出多个问题
- **优先选择题**：降低用户认知负担
- **YAGNI**：不需要的不做，不确定的不加
- **探索替代方案**：始终提供 2-3 种选择
- **设计隔离清晰**：
  - 拆分为职责单一的小模块
  - 定义清晰接口，可独立理解和测试
  - 小模块更容易推理和维护

## 在已有代码库中工作

- 先探索现有结构，再提出变更
- 遵循现有模式和约定
- 仅在发现明确问题时提出针对性改进
- 不提议无关的重构

## TDD 兼容性

如果用户要求 TDD，设计阶段需额外考虑：

- 模块边界是否清晰（便于单独测试）
- 依赖是否可注入（便于 mock）
- 接口是否明确（便于断言）
- 详见 [tdd.md](./tdd.md)
