# 执行开发

## 概述

执行开发阶段将计划转化为代码。根据任务间的依赖关系，选择合适的执行模式完成所有开发工作。

**核心原则：开发过程中绝不 git commit。**

---

## 执行模式

### Inline 顺序执行（有依赖）

当任务之间存在依赖关系时，按顺序逐个执行：

- 严格按照计划中的步骤执行
- 每个任务完成后运行验证
- 前一个任务的输出作为后续任务的输入

### Subagent 并行执行（无依赖）

当多个任务之间相互独立时，通过 subagent 并行执行：

- 每个任务分配独立的 subagent
- 每个 subagent 获得：明确的范围、清晰的目标、约束条件、期望输出
- **绝不让多个 subagent 编辑相同的文件**
- 按独立领域分组派发
- 所有 subagent 返回后，审查并整合结果，验证无冲突

---

## 执行策略

### 完整流程（有 plan 文件）

1. 读取 plan 文件，审查每个任务的依赖关系标注
2. 根据依赖关系决定执行方式：
   - **全部有依赖** → inline 顺序执行
   - **全部无依赖** → subagent 并行执行
   - **混合** → 先并行执行无依赖任务，再顺序执行有依赖任务

### 直接开发路径（无 plan 文件）

- 默认采用 inline 顺序执行
- 如果 agent 识别出可并行的独立任务，先向用户简述拆分方案，确认后再并行派发

---

## 使用 task 工具跟踪进度

用 `task` 工具管理整个执行过程：

```
# 初始化：使用中文任务名
task init ["任务一：创建数据模型", "任务二：实现 API 接口", "任务三：编写测试"]

# 开始任务时
task update index=0 status=in_progress

# 完成任务时
task update index=0 status=complete
```

- 初始化时使用中文任务名，简明描述任务内容
- 每个任务开始时立即标记为 `in_progress`
- 任务完成且验证通过后标记为 `complete`

---

## 单个任务执行流程

对每个任务：

1. **标记开始** — `task update` 设为 `in_progress`
2. **严格执行** — 按照计划中的步骤逐步执行
3. **运行验证** — 执行计划中指定的验证命令
4. **标记完成** — 验证通过后 `task update` 设为 `complete`

---

## Subagent 派发规范

为每个并行任务创建 subagent 时，提供：

- **范围**：明确告知可以修改哪些文件/目录
- **目标**：一句话描述要实现的功能
- **约束**：不得修改的文件、必须遵守的规范
- **期望输出**：完成后应产生的结果

subagent 返回后：

- 审查每个 subagent 的修改
- 确认修改之间无冲突
- 运行整体验证确保集成正确

---

## 规则

1. **绝不 git commit** — 开发过程中不执行任何 git 提交操作
2. **遇到阻塞立即停下** — 不猜测、不假设，向用户询问明确方向
3. **所有任务完成后进入 code review loop** — 不自行宣告完成
4. **严格遵循计划** — 不跳步骤、不省验证
5. **并行任务不得编辑相同文件** — 避免冲突
