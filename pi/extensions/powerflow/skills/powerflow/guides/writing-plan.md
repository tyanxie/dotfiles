# 编写开发计划

## 原则

假设执行者对项目零上下文。所有信息必须自包含：涉及哪些文件、具体代码、如何验证。
将计划拆分为可独立执行的小任务。DRY、YAGNI。

## 保存路径

```
docs/powerflow/YYYY-MM-DD-feature-name/plan.md
```

## 范围检查

如果需求涉及多个独立子系统，建议拆分为多份独立计划。

## 文件结构映射

在定义任务前，先列出所有将创建或修改的文件：

```markdown
## 文件结构

- 新建: src/feature/core.ts
- 新建: src/feature/types.ts
- 修改: src/index.ts
- 新建: tests/feature/core.test.ts（可选，TDD 模式下必须）
```

设计要求：

- 模块边界清晰，职责单一
- 倾向小而专注的文件
- 一起变更的文件放在一起

## 任务结构

```markdown
### 任务 N: [组件名称]

**依赖**: 任务 X, 任务 Y（或「无」）
**可并行**: 是/否

**文件:**

- 新建: exact/path/to/file
- 修改: exact/path/to/existing
- 测试: tests/exact/path/to/test（可选）

**步骤:**

1. 具体操作描述，包含实际代码
2. 验证方式：确切命令 + 预期输出
```

### 任务粒度

- 每个任务聚焦一个逻辑单元（一个函数、一个模块、一组紧密关联的变更）
- 步骤包含实际代码片段和确切的验证命令
- 预期输出必须明确写出

### 依赖关系标注

每个任务开头必须标注依赖和并行性：

```markdown
### 任务 1: 类型定义

**依赖**: 无
**可并行**: 是

### 任务 2: 核心逻辑

**依赖**: 任务 1
**可并行**: 是（与任务 3 可并行）

### 任务 3: UI 组件

**依赖**: 任务 1
**可并行**: 是（与任务 2 可并行）

### 任务 4: 集成

**依赖**: 任务 2, 任务 3
**可并行**: 否
```

## 无占位符原则

每个步骤必须包含完整可执行的内容。禁止出现以下写法：

- TBD / TODO
- 后续实现 / 稍后补充
- 添加适当的错误处理
- 类似任务 N 的做法
- 省略号代替实际代码

## TDD 模式（可选）

如果用户要求 TDD，每个任务的步骤需遵循 RED-GREEN-REFACTOR 周期：

```markdown
**步骤:**

1. RED - 编写失败测试
   - 测试代码（完整）
   - 运行命令: `npm test -- --filter "xxx"`
   - 预期: 测试失败，错误信息为 "xxx"

2. GREEN - 最小实现使测试通过
   - 实现代码（完整）
   - 运行命令: `npm test -- --filter "xxx"`
   - 预期: 测试通过

3. REFACTOR - 重构（如需要）
   - 重构内容描述 + 代码
   - 运行命令: `npm test`
   - 预期: 全部测试仍然通过
```

## 自检清单

计划编写完成后，逐项检查：

- [ ] 需求覆盖：所有需求点均有对应任务
- [ ] 占位符扫描：无 TBD/TODO/省略号
- [ ] 类型一致：跨任务的接口/类型定义一致
- [ ] 依赖关系：无循环依赖，拓扑顺序合理
- [ ] 验证完备：每个任务有明确的验证方式
